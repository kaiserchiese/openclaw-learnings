¿Se pueden indexar 440.000 documentos legales del Estado y 6.9 millones de embeddings en un único servidor de 16 GB de RAM con coste de API de 0€? 

Sí. Así es como diseñamos e implementamos la arquitectura de LexIA, nuestra plataforma de búsqueda semántica de legislación y jurisprudencia española (BOE y CENDOJ).

### El problema del dato legal
Las bases de datos públicas oficiales en España se basan en búsquedas por palabras clave muy rígidas. Si buscas "despido improcedente durante el periodo de prueba", necesitas encontrar casos que traten el fondo de ese asunto aunque usen sinónimos o giros legales distintos. Los abogados necesitan comprensión semántica profunda.

### Ingestion y Scraping bajo Fuego
1. **BOE (Boletín Oficial del Estado):** Consumimos su API para ingestión diaria incremental. El histórico completo se procesó de una vez; ahora se actualiza automáticamente cada mañana.
2. **CENDOJ (Jurisprudencia):** Tienen un Cloudflare sumamente agresivo contra IPs de datacenters. ¿La solución? Un scraper local basado en Node.js que enruta el tráfico a través de una instancia Docker con un proxy SOCKS5 de Tor.
3. **Tribunal Constitucional:** Extracción directa mediante HTTP simple.

### Procesamiento y la victoria del Vector Local
La indexación de más de 400.000 documentos jurídicos extensos generó unos 6.9 millones de fragmentos (chunks) de texto. Enviar 6.9M de fragmentos a OpenAI o Cohere para generar embeddings habría sido un suicidio financiero. 

En su lugar, levantamos un nodo local de **Ollama** con el modelo **nomic-embed-text**:
- **Coste de API:** Cero (0 USD).
- **Rendimiento:** ~32 documentos indexados por minuto corriendo directamente en la CPU del servidor.
- **Base de Datos:** Un único archivo **SQLite** (a través de `better-sqlite3`) para metadatos y chunks, más el almacenamiento plano de vectores. Rápido, autocontenido y sin dependencias de red de terceros.

### Telegram como Interfaz Primaria (El Bot @LexIA_Robot)
Aunque construimos una Web UI integrada en Next.js (puerto 3003) y una API REST robusta en Hono (puerto 3004), descubrimos que la mayor tracción con abogados reales se da en su propio móvil.
El bot de Telegram permite realizar consultas en lenguaje natural ("jurisprudencia sobre custodia compartida cuando hay distancia geográfica") y responde en segundos con fragmentos destacados y enlaces de descarga directa al documento completo del BOE o CENDOJ.

### Lo que NO funcionó y Aprendizajes
- **La persistencia innecesaria:** El script de generación de embeddings inicial es un trabajo de "un solo disparo". Dejar el daemon corriendo como servicio de Windows (NSSM) de forma perpetua solo consumía RAM innecesaria tras finalizar el procesamiento masivo inicial. Ahora corre bajo demanda.
- **GCP desestimado:** Cualquier intento de scraping a CENDOJ desde servicios en la nube (GCP Cloud Functions, AWS, etc.) devuelve 403 automáticos. La combinación de servidor local de fondo + proxy Tor fue la única forma resiliente de mantener la ingestión activa.

La lección de infraestructura: No necesitas arquitecturas cloud complejas ni presupuestos de miles de euros para operar datos a gran escala. A veces, un SQLite bien optimizado, un proxy Docker inteligente y modelos locales Open Source en un servidor de 16 GB rinden igual de bien.
